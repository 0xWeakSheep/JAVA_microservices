### 任务4：验证Zuul前置过滤器的优先级顺序

**实现内容**：
- 创建了3个自定义 Zuul 前置过滤器（Pre Filter）
- 分别设置不同的 `filterOrder` 值验证优先级
- 过滤器按 order 值从小到大顺序执行（order 越小优先级越高）

**创建的过滤器**：

| 过滤器名称 | Order 值 | 优先级 | 功能说明 |
|------------|----------|--------|----------|
| `LoggerPreFilter` | 1 | 最高 | 记录请求日志（时间、方法、URI、IP） |
| `AuthPreFilter` | 2 | 中等 | 进行权限校验，检查Authorization头 |
| `ParamCheckPreFilter` | 3 | 最低 | 参数校验，检查非法字符 |

**过滤器代码示例**：

**LoggerPreFilter.java** (Order = 1)：
```java
@Component
public class LoggerPreFilter extends ZuulFilter {
    private static final Logger logger = LoggerFactory.getLogger(LoggerPreFilter.class);

    @Override
    public String filterType() {
        return "pre";  // 前置过滤器
    }

    @Override
    public int filterOrder() {
        return 1;  // 最高优先级
    }

    @Override
    public boolean shouldFilter() {
        return true;
    }

    @Override
    public Object run() {
        RequestContext ctx = RequestContext.getCurrentContext();
        HttpServletRequest request = ctx.getRequest();

        logger.info("=== [LoggerPreFilter] 优先级1 - 请求日志记录 ===");
        logger.info("请求时间: {}", LocalDateTime.now());
        logger.info("请求方法: {}", request.getMethod());
        logger.info("请求路径: {}", request.getRequestURI());

        // 将时间戳存入上下文
        ctx.set("startTime", System.currentTimeMillis());
        return null;
    }
}
```

**AuthPreFilter.java** (Order = 2)：
```java
@Component
public class AuthPreFilter extends ZuulFilter {
    private static final Logger logger = LoggerFactory.getLogger(AuthPreFilter.class);

    @Override
    public String filterType() {
        return "pre";
    }

    @Override
    public int filterOrder() {
        return 2;  // 中等优先级
    }

    @Override
    public boolean shouldFilter() {
        return true;
    }

    @Override
    public Object run() {
        RequestContext ctx = RequestContext.getCurrentContext();
        HttpServletRequest request = ctx.getRequest();

        logger.info("=== [AuthPreFilter] 优先级2 - 认证检查 ===");

        String token = request.getHeader("Authorization");
        String uri = request.getRequestURI();

        // 放行公开路径
        if (uri.contains("/login") || uri.contains("/register")) {
            logger.info("公开路径，跳过认证: {}", uri);
            return null;
        }

        if (token == null || token.isEmpty()) {
            logger.warn("认证失败: 未找到Authorization请求头");
            ctx.setSendZuulResponse(false);
            ctx.setResponseStatusCode(401);
            ctx.setResponseBody("{\"error\": \"未授权访问\"}");
        } else {
            logger.info("认证通过");
            ctx.set("authToken", token);
        }
        return null;
    }
}
```

**ParamCheckPreFilter.java** (Order = 3)：
```java
@Component
public class ParamCheckPreFilter extends ZuulFilter {
    private static final Logger logger = LoggerFactory.getLogger(ParamCheckPreFilter.class);

    @Override
    public String filterType() {
        return "pre";
    }

    @Override
    public int filterOrder() {
        return 3;  // 最低优先级
    }

    @Override
    public boolean shouldFilter() {
        return true;
    }

    @Override
    public Object run() {
        RequestContext ctx = RequestContext.getCurrentContext();
        HttpServletRequest request = ctx.getRequest();

        logger.info("=== [ParamCheckPreFilter] 优先级3 - 参数校验 ===");

        // 获取之前过滤器存储的信息
        Long startTime = (Long) ctx.get("startTime");
        if (startTime != null) {
            long elapsedTime = System.currentTimeMillis() - startTime;
            logger.info("过滤器链执行耗时: {} ms", elapsedTime);
        }

        // 参数校验
        String param = request.getParameter("name");
        if (param != null && (param.contains("<") || param.contains(">"))) {
            logger.warn("参数校验失败: 检测到非法字符");
            ctx.setSendZuulResponse(false);
            ctx.setResponseStatusCode(400);
            ctx.setResponseBody("{\"error\": \"请求参数不合法\"}");
            return null;
        }

        logger.info("所有前置过滤器执行完成，准备转发请求");
        return null;
    }
}
```

### 遇到的问题与解决

| 问题 | 解决方案 |
|------|----------|
| Zuul 过滤器不生效 | 确保过滤器类添加了 `@Component` 注解被 Spring 管理 |
| 过滤器执行顺序不符合预期 | 检查 `filterOrder()` 返回值，数值越小优先级越高 |
| 请求上下文传递失败 | 使用 `RequestContext.getCurrentContext()` 获取上下文 |

### 实验总结

通过本次实验，深入理解了 Spring Cloud Zuul 网关在微服务架构中的核心作用。Zuul 作为微服务架构的统一入口，承担着路由转发、权限校验、日志记录等重要职责。通过搭建 Eureka 注册中心、Zuul 网关和 dept-service 微服务，构建了一个完整的微服务调用链路。

在过滤器实验中，创建了3个具有不同优先级的自定义前置过滤器（LoggerPreFilter、AuthPreFilter、ParamCheckPreFilter），通过设置不同的 `filterOrder` 值（1、2、3），验证了 Zuul 过滤器按 Order 值从小到大顺序执行的机制。Order 值越小，过滤器优先级越高，越早执行。这一机制使得开发者可以精确控制过滤器的执行顺序，实现如先记录日志、再校验权限、最后统计访问等有序处理流程。

通过本次实践，掌握了 Zuul 过滤器的开发方法，理解了过滤器在请求处理流程中的位置，为后续实现更复杂的网关功能（如统一认证、限流熔断、灰度发布等）打下了坚实基础。

```yaml
    name: ribbon-consumer

  client:
    service-url:
      defaultZone: http://localhost:8761/eureka/

dept-service-provider:
  ribbon:
    NFLoadBalancerRuleClassName: com.netflix.loadbalancer.RoundRobinRule
    ConnectTimeout: 1000
    ReadTimeout: 3000
```

**启动类代码**：
```java
@SpringBootApplication
@EnableDiscoveryClient
public class RibbonConsumerApplication {

    public static void main(String[] args) {
        SpringApplication.run(RibbonConsumerApplication.class, args);
    }

    @Bean
    @LoadBalanced
    public RestTemplate restTemplate() {
        return new RestTemplate();
    }
}
```

**ConsumerController代码**：
```java
@RestController
@RequestMapping("/consumer")
public class ConsumerController {

    @Autowired
    private RestTemplate restTemplate;

    private static final String SERVICE_URL = "http://dept-service-provider";

    @GetMapping("/test")
    public Map testLoadBalance() {
        List<Map> results = new ArrayList<>();
        
        // 连续调用10次服务
        for (int i = 0; i < 10; i++) {
            Map result = restTemplate.getForObject(
                SERVICE_URL + "/dept/info", Map.class);
            results.add(result);
        }

        Map response = new HashMap();
        response.put("totalCalls", 10);
        response.put("results", results);
        response.put("strategy", "RoundRobinRule (轮询)");
        
        return response;
    }
}
```

## 五、Ribbon支持的负载均衡策略

| 策略类 | 名称 | 说明 |
|--------|------|------|
| RoundRobinRule | 轮询 | 按顺序轮流选择（默认） |
| RandomRule | 随机 | 随机选择一个实例 |
| RetryRule | 重试 | 先轮询，失败后在指定时间内重试 |
| WeightedResponseTimeRule | 响应时间加权 | 根据响应时间分配权重 |
| BestAvailableRule | 最优可用 | 选择并发请求最少的实例 |
| AvailabilityFilteringRule | 可用过滤 | 过滤掉故障实例，然后轮询 |

## 六、实验总结

### 完成的功能

1. ✅ **Eureka Server 正常运行** - 端口 8761
2. ✅ **5个同名微服务实例** - 端口 8001-8005，全部注册到Eureka
3. ✅ **Ribbon Consumer 搭建成功** - 端口 9001，使用RestTemplate+@LoadBalanced
4. ✅ **负载均衡验证成功** - 轮询策略生效，请求均匀分发到5个实例

### 实验原理

Ribbon 是 Netflix 开源的客户端负载均衡器，工作原理：
1. 从 Eureka 获取服务注册列表
2. 根据配置的负载均衡策略（如轮询）选择服务实例
3. 将请求发送到选中的实例
4. 客户端维护负载均衡状态，不依赖服务端

### 实验总结

通过本次实验，深入理解了 Spring Cloud Ribbon 客户端负载均衡的工作原理。通过搭建5个同名的 `dept-service-provider` 微服务实例，并使用 `ribbon-consumer` 通过服务名调用，验证了 Ribbon 默认的轮询负载均衡策略。

实验结果显示，10次连续调用被均匀分配到5个实例上，每个实例处理2次请求，严格按照轮询顺序（8001→8002→8003→8004→8005→8001...）执行。这证明了 Ribbon 能够有效地将请求分发到多个服务实例，提高系统的可用性和吞吐量。
