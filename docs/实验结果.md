### 任务3：实现配置的动态刷新（无需重启服务）

**实现内容**：
- 在 `JwtService` 和 `ConfigDemoController` 上添加 `@RefreshScope` 注解
- 配置 Actuator 暴露 `/actuator/refresh` 端点
- 修改 Git 仓库配置后，通过 POST 请求刷新配置

**验证步骤与结果**：

**步骤1**：查看当前配置
```bash
curl http://localhost:3001/api/config/info
```
结果：`app.description` = "开发环境"

**步骤2**：修改 Git 仓库配置文件
```bash
cd ~/.config-repo/movies-service/
vim movies-service-dev.yml
# 修改 description: 开发环境 -> 开发环境-已刷新
```

**步骤3**：提交到 Git 仓库
```bash
git add .
git commit -m "更新配置测试动态刷新"
```

**步骤4**：发送刷新请求（无需重启服务）
```bash
curl -X POST http://localhost:3001/actuator/refresh
```
结果：`["app.description"]` - 表示该配置已更新

**步骤5**：验证配置已刷新
```bash
curl http://localhost:3001/api/config/info
```
结果：`app.description` = "开发环境-已刷新" ✅

## 五、实验总结

### 完成的功能

1. ✅ **Config Server 搭建成功** - 端口 8888，可对外提供配置接口
2. ✅ **Config Client 改造成功** - 业务服务从远程拉取配置启动
3. ✅ **动态刷新功能正常** - 修改配置后无需重启服务，通过 `/actuator/refresh` 刷新
4. ✅ **多环境配置切换正常** - dev/test/prod 三个环境配置完整，切换灵活

### 遇到的问题与解决

| 问题 | 解决方案 |
|------|----------|
| Config Server 无法读取本地 Git 仓库 | 确保 Git 仓库已初始化并有提交记录 |
| 客户端启动时连接 Config Server 失败 | 添加 `optional:` 前缀允许失败回退 |
| @Value 注入的配置无法动态刷新 | 使用 `@RefreshScope` 注解 |

### 实验总结

通过本次实验，深入理解了分布式配置中心在微服务架构中的核心作用，掌握了 Spring Cloud Config Server 的搭建与使用方法，能够独立完成配置中心的部署和 Git 仓库集成。同时，成功将业务服务改造为 Config Client，实现了从远程配置中心动态拉取配置并启动服务。通过引入 @RefreshScope 注解和 Actuator 端点，实现了配置的热更新功能，无需重启服务即可使配置变更生效。此外，还完成了多环境配置管理，能够灵活切换 dev、test、prod 三种环境的配置，为微服务项目的配置管理提供了标准化解决方案。通过本次实践，对 Spring Cloud 生态中的配置管理组件有了更深入的理解和实践经验。
